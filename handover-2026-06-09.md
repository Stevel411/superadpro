## 🎓 KEY LEARNINGS (carry forward)

- **Stripe:** use `Charge.list()`, not `BalanceTransaction.list(type=...)` — Checkout/sub payments are `payment`-typed `py_` ids. `PaymentIntent.list()` throws `KeyError:0` in this env. Use `.to_dict()` on Stripe objects.
- **NOWPayments:** `get_payment_status(id)` works with `x-api-key`; **list-payments (`GET /v1/payment`) needs a JWT** from `POST /v1/auth` (email+password). Env keys: `NOWPAYMENTS_EMAIL`, `NOWPAYMENTS_PASSWORD` (added this session). Forwarding hot wallet: `0xa96be652…becd09`. `distinct_uids_seen` = order existence, NOT proof of payment (includes `expired`/`$0`).
- **BSC is PoA:** web3 `eth.get_block` raises `ExtraDataLengthError` without PoA middleware → use raw `eth_getBlockByNumber` JSON-RPC for timestamps. getLogs is unaffected. Endpoint runs on Railway using existing `BSC_RPC_URL` (Alchemy) — no new key needed. Public keyless RPCs are dead for this (dataseed rejects ≥50-block getLogs; Ankr needs a key; BSCScan API now Etherscan-unified/paid).
- **WalletConnect unique amounts:** `base ± up to 50¢` (`UNIQUE_RANGE_CENTS=50`), deterministic from `(user_id, order_id)`. Founding band $14.50–$15.50. Order_id wiped → exact recompute impossible → **post-wipe direct payments cannot be provably attributed to a member.**
- **member_composition "comped"** = "no surviving payment record." Off-rail payment channels that leave no standard rail row: bank transfer, off-coin NOWPayments (ETH misfund), Free Gifting (only this one is DB-queryable via `GiftVoucher`).
- **On-chain matching:** TIGHT time windows essential. Wide windows + unapplied dry-run matches cause one tx to be claimed by multiple members. Verify no tx_hash appears for two members before apply.
